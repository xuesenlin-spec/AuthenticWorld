\begin{abstract}
Despite remarkable progress in mobile GUI agents, existing approaches predominantly rely on myopic step-by-step decision-making, which fails to handle long-horizon tasks (≥16 steps), cross-app workflows, and conditional branches effectively. These methods suffer from catastrophic forgetting, error propagation, and exponential context cost as task length increases. In this paper, we propose a \textbf{hierarchical long-horizon planning architecture} for Android GUI agents that integrates explicit global planning, structured state memory, and dynamic backtracking mechanisms. \section{INTRODUCTION}
Mobile GUI agents have emerged as a promising paradigm for automating smartphone tasks, with recent work achieving impressive performance on benchmarks like AndroidWorld \cite{androidworld2024}. However, current state-of-the-art agents, including DroidRun \cite{droidrun2026}, K²-Agent \cite{k2agent2026}, and Minitap \cite{minitap2026}, adopt a myopic decision-making paradigm where each action is chosen based only on the current interface and a short history of recent steps. This approach works well for simple tasks but fails catastrophically when faced with long-horizon, cross-app, or multi-branch workflows.

We identify four critical limitations of existing GUI agents:
\begin{enumerate}
    \item \textbf{Myopic Decision-Making}: Lack of explicit modeling of global task dependencies, branch logic, and long-term goals, leading to goal forgetting and path deviation in long sequences.
    \item \textbf{Context Cost Explosion}: Concatenating raw screenshots or a11y trees for long tasks causes token overflow, increased inference latency, and hallucinations.
    \item \textbf{Irreversible Error Propagation}: Minor early errors (e.g., misclicks, incorrect text input) amplify over time, with no mechanism for global backtracking or recovery.
    \item \textbf{Poor Cross-App/Asynchronous Adaptation}: Random popups, loading delays, and app switches disrupt execution, as myopic agents cannot anticipate or dynamically adjust long-term plans.
\end{enumerate}

To address these limitations, we present a hierarchical long-horizon planning architecture for mobile GUI agents. Our key contributions are:
\begin{enumerate}
    \item A three-level hierarchical planning framework that explicitly models long-range dependencies and conditional branches, solving the goal forgetting problem in myopic approaches.
    \item A GUI-specific structured memory module that stores critical state information instead of raw data, reducing context overhead by \textbf{[TO BE FILLED]}\% compared to full history concatenation.
    \item A dynamic replanning and breakpoint backtracking system that detects execution anomalies and revises plans at runtime, blocking error propagation.
    \item The AndroidWorld-Long benchmark, an extended evaluation set for long-horizon GUI tasks, providing a standardized testbed for future research.
\end{enumerate}

\section{RELATED WORK}
\subsection{Benchmark for GUI Agents}
Recent years have witnessed the emergence of several benchmarks for evaluating GUI agents on mobile devices. Notable
  examples include AndroidWorld~\cite{rawles2024androidworld}, which provides a suite of 116 tasks across 16 Android
  applications; AITW~\cite{rawles2023androidinthewild}, a large-scale dataset of human-device interactions; and
  WebShop~\cite{yao2022webshop}, which focuses on web-based e-commerce navigation. While these benchmarks have
  established foundational evaluation protocols, we identify several critical limitations that hinder their ability to
  rigorously assess long-horizon GUI agent capabilities.

  Limitations of existing benchmarks. First, \textit{task horizon is too short}. The majority of tasks in AndroidWorld
  and similar benchmarks require fewer than 15 steps to complete, which fails to expose the compounding error
  accumulation that characterizes real-world long-horizon interactions. Second, \textit{cross-app interactions are
  limited}. Most tasks are confined to a single application or involve at most two apps in a simple sequential pattern
  (e.g., toggle a system setting then open an app). Real user workflows, however, frequently involve data flowing
  through three or more applications in complex pipelines. Third, \textit{lack of conditional decision-making}. Existing
   tasks follow predetermined action sequences with no branching logic based on the current device state. In contrast,
  real users constantly make context-dependent decisions—checking whether WiFi is enabled before proceeding, deleting
  duplicate calendar events before creating new ones, or determining whether to add or remove items based on existing
  content. Fourth, \textit{scenarios lack authentic user motivation}. Many benchmark tasks appear mechanically
  constructed (e.g., "check if note count exceeds threshold, then delete oldest notes") rather than reflecting genuine
  user goals. This artificial construction limits the ecological validity of the evaluation and may not reveal failure
  modes that agents encounter in realistic usage patterns.

  To address these gaps, we design a new benchmark that evaluates GUI agents on \textit{long-horizon} (50+ steps),
  \textit{multi-app} (3+ applications) tasks grounded in authentic user scenarios. Our benchmark organizes tasks into
  four categories, each targeting a distinct capability dimension:
  
  \textbf{(1) Cross-App Information Flow.} These tasks evaluate an agent's ability to extract information from one
  application, transform it, and utilize it as input for subsequent applications. Examples include
  \textit{BusinessTripPlanning} (extract travel dates from calendar, record destination coordinates in Maps, compose a
  trip itinerary note, and notify colleagues via SMS), \textit{ExpenseReimbursement} (read expense details from notes,
  enter them into an expense tracker, and send a summary to the finance department), and \textit{PartyPlanning} (create
  calendar events, manage guest contacts, draft shopping lists, and send invitations). These tasks test the agent's
  capacity to maintain information integrity across application boundaries—a fundamental requirement for real-world
  productivity workflows.

  \textbf{(2) Conditional Decision-Making and Error Handling.} These tasks require agents to inspect the current device
  state and choose appropriate actions based on observed conditions. Examples include \textit{NetworkTroubleshooting}
  (check WiFi and Bluetooth states, enable/disable as needed, and log the actions taken),
  \textit{CalendarConflictResolution} (check if events already exist on a given date, then either consolidate or create
  new events accordingly), and \textit{BudgetCheckBeforePurchase} (verify whether cumulative expenses exceed a budget
  limit before approving new purchases). These tasks reveal whether agents can reason about state-dependent branching
  logic—a capability essential for robust autonomous operation but largely untested by existing benchmarks.

  \textbf{(3) Multi-Subgoal Workflows.} These tasks present a high-level objective composed of multiple interdependent
  subgoals, each targeting a different application. Examples include \textit{NewJobOnboarding} (add HR contacts, create
  a weekly work schedule, draft a work-info note, and send a confirmation message), \textit{WeeklyMealPrep} (plan a
  5-day menu, record grocery costs, schedule meal-time reminders, and share the menu with family), and
  \textit{HomeRenovationProject} (manage a task checklist, schedule contractor visits, track renovation expenses, and
  coordinate with workers). These tasks evaluate the agent's ability to decompose a complex goal, track progress across
  multiple independent subtasks, and synthesize a coherent final outcome.

  \textbf{(4) Long-Horizon Interaction with Exception Handling.} These tasks impose extended interaction sequences
  (70–90 steps) that require agents to navigate permission dialogs, system prompts, UI transitions, and potential state
  changes over time. Examples include \textit{TravelItineraryManagement} (schedule a multi-day trip across several
  cities, mark locations on a map, draft a detailed itinerary, and coordinate with travel companions),
  \textit{EventPlanningAndCoordination} (manage dozens of participants, send personalized invitations, track event
  budgets, and maintain a master planning document), and \textit{SmallBusinessDailyOperations} (record daily sales,
  manage customer contacts, schedule next-day tasks, and send business summaries to partners). These tasks are designed
  to stress-test the agent's resilience to real-world friction—permission requests that interrupt workflow, UI elements
  that load asynchronously, and the accumulation of minor errors over extended sequences.

  In total, our benchmark comprises 20 tasks spanning 4 categories, with step budgets ranging from 50 to 90 steps and
  requiring interactions across 3 to 6 applications per task. Each task is designed from a realistic user scenario to
  ensure ecological validity, and each targets specific failure modes—information loss across apps, inability to branch
  on state, failure to decompose complex goals, and error accumulation under extended interaction—that we hypothesize
  are critical bottlenecks for current GUI agents.



\subsection{Mobile GUI Agents}
Mobile GUI agent research has evolved along two primary paths: accessibility tree-based approaches and pure vision-based approaches. Accessibility tree methods \cite{droidrun2026, minitap2026} leverage structured UI semantic data, achieving high accuracy on AndroidWorld but lacking generality to environments without a11y support. Pure vision approaches \cite{k2agent2026, uitars2025} rely solely on screenshots, offering broader applicability but lower accuracy due to visual noise and localization errors. All existing methods, however, use myopic step-by-step decision-making, limiting their performance on long-horizon tasks.

\subsection{Long-Horizon Planning for LLM Agents}
Long-horizon planning has been extensively studied in general LLM agents \cite{hierarchicalplanning2023, memoryagents2024}. Hierarchical planning approaches \cite{llmplanning2023} decompose complex tasks into subtasks, while memory mechanisms \cite{longtermmemory2024} store critical information to reduce context overhead. However, these methods are not optimized for the unique characteristics of GUI tasks, such as dynamic UI changes, asynchronous events, and precise action localization requirements. Our work adapts and extends these general planning techniques to the mobile GUI domain.

\section{PRELIMINARY}
\subsection{AndroidWorld Benchmark}
AndroidWorld \cite{androidworld2024} is a dynamic, reproducible benchmark for Android GUI agents consisting of 116 tasks across 20 real-world apps. Tasks are categorized into Easy (1–5 steps), Medium (6–15 steps), and Hard (≥16 steps) based on complexity. The environment provides both screenshots and accessibility trees as input, and success is determined via system-level state verification (e.g., database queries, file system checks).

\subsection{Long-Horizon GUI Task Definition}
We define a long-horizon GUI task as a task requiring \textbf{≥16 sequential actions} to complete, with at least one of the following characteristics:
\begin{itemize}
    \item Cross-app interaction (switching between ≥2 apps)
    \item Conditional branch logic (if-else decision points)
    \item Long-term state dependencies (earlier actions affect later steps)
    \item Asynchronous events (loading delays, popups)
\end{itemize}

\subsection{Basic Concepts}
\begin{itemize}
    \item \textbf{Accessibility Tree (a11y Tree)}: A structured representation of the UI containing element roles, names, bounds, states, and available actions.
    \item \textbf{Atomic Action}: A single executable operation, including click, long click, text input, scroll, and app switch.
    \item \textbf{Checkpoint}: A verified state in the execution process that serves as a valid recovery point for backtracking.
\end{itemize}

\section{METHODOLOGY}
\subsection{Overall Architecture}
Our architecture consists of four core modules that operate in a hierarchical, decoupled manner:
\begin{enumerate}
    \item \textbf{Global Long-Horizon Planner}: Generates a structured global execution plan from the task description.
    \item \textbf{Structured State Memory Module}: Stores critical state information and execution history with dynamic eviction.
    \item \textbf{Local Executor}: Executes individual subtasks using either a11y tree or pure vision input.
    \item \textbf{Anomaly Detector \& Dynamic Replanner}: Monitors execution, detects anomalies, and triggers backtracking or plan revision.
\end{enumerate}

The execution pipeline is as follows:
\[
\text{Task Instruction} \rightarrow \text{Global Planner} \rightarrow \text{Subtask Sequence} \rightarrow \text{Memory + Local Executor} \rightarrow \text{Atomic Action} \rightarrow \text{Device Execution} \rightarrow \text{Anomaly Detector} \rightarrow \text{Replanning/Backtracking}
\]


\subsection{REFLECT: Loop-Aware Self-Correction via LLM-Driven Reflection}
  \label{subsec:reflect}

  Autonomous GUI agents operating in long-horizon settings inevitably encounter
  failure modes such as repetitive action loops, transient system dialogs, and
  stuck states. Recent agents have introduced reflection mechanisms to address
  these issues, but they exhibit critical limitations when applied to extended
  interaction sequences. We first analyze existing approaches, then present our
  REFLECT mechanism and its advantages.

  \subsubsection{Analysis of Existing Reflection Mechanisms}

  \paragraph{MiniTap.} MiniTap~\cite{minitap} employs a post-hoc reflection
  strategy in which the agent reviews the entire execution history after task
  completion or timeout. The reflection prompt contains the full action history
  and asks the LLM to identify what went wrong and propose corrections. While
  this enables global reasoning, it suffers from two fundamental drawbacks:
  (i)~reflection occurs too late---by the time the agent identifies the error
  pattern, the step budget has typically been exhausted; and (ii)~injecting the
  entire history as context consumes substantial token budget, degrading LLM
  reasoning quality as the context window fills.

  \paragraph{K2Agent.} K2Agent~\cite{k2agent} uses an inline error-detection
  mechanism that triggers when the agent's predicted UI action fails to produce
  the expected accessibility tree state change. This is more timely than post-hoc
  reflection, as it detects failures within a single step. However, it relies on
  a rigid rule-based comparator (predicted vs.\ observed UI state) that cannot
  handle ambiguous failures such as transient system dialogs that appear and
  disappear within 2--5~seconds. Additionally, K2Agent's corrective action is
  limited to retrying the same operation with a different target element; it
  lacks the capacity to propose fundamentally different strategies.

  \paragraph{Summary of Limitations.} Three common weaknesses emerge across
  existing approaches:
  (1)~\textit{Single-shot reflection.} Once the agent executes a corrective
  action, there is no mechanism to evaluate whether the new strategy is working,
  leading to secondary loops.
  (2)~\textit{No time-sensitive action support.} Existing agents cannot respond
  to transient UI elements (e.g., system dialogs) because the LLM reasoning
  latency (typically 10--50~s) exceeds the dialog's lifetime.
  (3)~\textit{Unstructured history injection.} Reflection results are appended
  as plain text to the prompt, making it difficult for the LLM to distinguish
  between user actions, system events, and reflection insights.

  \subsubsection{The PegaAgent REFLECT Mechanism}

  Our REFLECT mechanism addresses these limitations through three design
  principles that separate \textit{detection}, \textit{diagnosis}, and
  \textit{intervention} into distinct, composable modules.

  \paragraph{Proactive loop detection with deferred notification.}
  Rather than waiting for task completion, REFLECT monitors a sliding window of
  recent $(\text{action}, \text{target})$ pairs at every step. When the same
  action repeats $N$ times (default $N=3$) without meaningful state progression,
  it flags a loop condition. Instead of immediately executing a corrective
  action---which may itself be incorrect---REFLECT appends a structured loop
  notification to the next LLM prompt, preserving the agent's autonomy to choose
  the most appropriate recovery strategy.

  \paragraph{Two-mode Fast-Path as a reflection action.}
  REFLECT introduces \texttt{fast\_path} as a first-class action type that the
  LLM can invoke during reflection. It supports two modes:
  (i)~\texttt{immediate}---execute a button click on a currently visible system
  dialog with $<\!1$~s latency, bypassing the LLM;
  (ii)~\texttt{next\_step}---set a deferred handler that will automatically
  check for and handle a dialog \textit{after} the next action executes. The
  second mode is essential when the dialog only appears as a \textit{consequence}
  of an action (e.g., clicking ``Send'' triggers a ``set default SMS app''
  prompt that disappears before the LLM can respond).

  \paragraph{One-shot consumption with structured summary injection.}
  After a deferred Fast-Path triggers, it is immediately consumed (one-shot
  semantics). The result is injected into the step summary as a structured tag
  (e.g., \texttt{[FAST-PATH deferred] Dialog handled. Clicked 'CHANGE'}),
  ensuring the LLM sees both the original action and the system's automatic
  handling in the subsequent prompt. This structured format prevents the LLM
  from conflating system interventions with its own decisions.

  \subsubsection{Algorithm}

  Algorithm~\ref{alg:reflect} formalizes the REFLECT mechanism. The key insight
  is the separation of detection (rule-based, $O(1)$), diagnosis (LLM-driven),
  and intervention (potentially time-sensitive). This decomposition enables the
  agent to handle failure modes that monolithic reflection mechanisms cannot
  address.


  \begin{algorithm}[htbp]
  \caption{PegaAgent REFLECT Step}
  \label{alg:reflect}
  \begin{algorithmic}[1]
  \Require Goal $G$, history $\mathcal{H}$, loop detector $L$, deferred fast-path $D \gets \text{None}$
  \Ensure Step result $R$

  \State $state \gets \Call{GetPostTransitionState}{}$
  \State $UI \gets state.\text{ui\_elements}$

  \Statex \textbf{// Loop Detection \& Notification}
  \State $notify \gets \text{``''}$
  \If{$\Call{IsStuck}{L}$}
      \State $problem \gets \Call{GetProblemDescription}{L}$
      \State $notify \gets \Call{BuildLoopPrompt}{problem}$
  \EndIf

  \Statex \textbf{// LLM Decision}
  \State $prompt \gets \Call{BuildActionPrompt}{G, \mathcal{H}, UI} + notify$
  \State $(reason, action\_json) \gets \Call{LLM.predict}{prompt}$
  \State $a \gets \Call{ParseJSON}{action\_json}$

  \Statex \textbf{// Handle fast\_path Action}
  \If{$a.\text{action\_type} = \text{``fast\_path''}$}
      \If{$a.\text{mode} = \text{``immediate''}$}
          \If{$\Call{HasDialog}{UI}$}
              \State $btn \gets \Call{FindButton}{UI, a.\text{button\_text}}$
              \If{$btn \neq \text{None}$}
                  \State $\Call{ExecuteClick}{btn}$
                  \State $state \gets \Call{GetPostTransitionState}{}$
                  \State $UI \gets state.\text{ui\_elements}$
              \EndIf
          \EndIf
          \State $(reason, action\_json) \gets \Call{LLM.predict}{\text{rebuild prompt with new } UI}$
          \State $a \gets \Call{ParseJSON}{action\_json}$
      \ElsIf{$a.\text{mode} = \text{``next\_step''}$}
          \State $D \gets a.\text{button\_text}$
          \State $\Call{AppendHistory}{\mathcal{H}, \text{``[FAST-PATH] Deferred''}}$
          \State \Return $(done \gets \text{false})$
      \EndIf
  \EndIf

  \Statex \textbf{// Execute Normal Action}
  \State $act \gets \Call{ConvertToJSONAction}{a}$
  \State $\Call{ExecuteAction}{env, act}$
  \State $\Call{Record}{L, act.\text{action\_type}}$

  \Statex \textbf{// Check Deferred Fast-Path}
  \If{$D \neq \text{None}$}
      \State $state \gets \Call{GetPostTransitionState}{}$
      \If{$\Call{HasDialog}{state.\text{ui\_elements}}$}
          \State $btn \gets \Call{FindButton}{state.\text{ui\_elements}, D}$
          \If{$btn \neq \text{None}$}
              \State $\Call{ExecuteClick}{btn}$
              \State $state \gets \Call{GetPostTransitionState}{}$
              \State $fp\_summary \gets \text{``[FAST-PATH] Clicked ''} + btn$
          \EndIf
      \EndIf
      \State $D \gets \text{None}$
  \EndIf

  \Statex \textbf{// Finalize Step}
  \State $summary \gets \Call{FormatSummary}{action\_json, reason, fp\_summary}$
  \State $\Call{AppendHistory}{\mathcal{H}, summary}$
  \State \Return $(done \gets \text{false})$
  \end{algorithmic}
  \end{algorithm}

\subsection{Long-chain Implicit Decision Mechanism}
\label{subsec:lidm}

\subsubsection{Motivation}

Long-horizon GUI agent tasks often require cross-app reasoning where critical
decision-making information is \textit{implicit}---not stated in the task goal,
but scattered across files, screens, or app states that the agent must
discover and read on its own. We define such problems as
\textbf{Long-chain Implicit Decision Problems} (LIDPs): tasks where the agent
must (1) navigate multiple apps to gather implicit information, (2) reason over
this information using rules that are also not part of the goal, and (3) make a
correct downstream decision based on the gathered facts.

Consider the \textit{MedicalAppointmentWorkflow} task: the agent must read a
symptom note, read a clinic booking policy (both hidden from the goal), count
the symptoms, and decide whether to book a ``Specialist Appointment''
($>3$ symptoms) or a ``Routine Checkup'' ($\leq 3$ symptoms). In our
experiments, baseline agents fail at Step 18 (filling the appointment title)
because the symptom count extracted at Step 10 is lost by the time the
decision is made. The root cause is that the agent's memory consists solely
of natural-language history summaries, which discard structured facts during
compression.

\subsubsection{Design Principles}

The mechanism must satisfy three constraints:
\begin{enumerate}[leftmargin=*]
    \item \textbf{Generality.} The mechanism must not rely on task-specific
    schemas or hand-crafted extraction rules. It should work across all LIDPs
    in the benchmark.
    \item \textbf{Timeliness.} Information must be captured \textit{when it is
    visible}, not retrieved retroactively after it is lost.
    \item \textbf{Goal-grounded.} What to remember should be determined by the
    agent's own reasoning about which facts will be needed later to achieve
    the goal.
\end{enumerate}

\subsubsection{Mechanism Overview}

We introduce the \textbf{Long-chain Implicit Decision Mechanism} (LIDM),
an explicit working memory system that operates in three phases at every
agent step:

\begin{enumerate}[leftmargin=*,noitemsep,topsep=2pt]
    \item \textbf{Goal-Aware Memory Prompt (GAMP).} Injected into the action
    selection prompt, instructing the LLM to identify and record facts from the
    current screen that will be needed later.
    \item \textbf{Working Memory Context (WMC).} Injects all previously recorded
    facts into the prompt so the agent can use them for downstream decisions.
    \item \textbf{Fact Parsing and Storage (FPS).} Parses the LLM's structured
    memory output and persists it into a \texttt{WorkingMemory} store.
\end{enumerate}

\paragraph{Goal-Aware Memory Prompt.}
At every step $t$, the action prompt $P_t^{\text{action}}$ is extended with:

\begin{lstlisting}[basicstyle=\small\ttfamily, breaklines=true,
                   frame=single, xleftmargin=10pt]
--- Goal-Aware Memory ---
Your overall goal is: {goal}
Look at the current screen carefully. Based on your goal,
is there any specific information here (numbers, names,
dates, file contents, rules, or decisions) that you will
need LATER to complete the goal?
If yes, record it using: [REMEMBER: key=value, key2=value2]
If nothing needs to be remembered, you can skip this.
\end{lstlisting}

The LLM answers this inline, before producing its action. Crucially, this
happens \textit{while the information is still visible on screen}, not
after the agent navigates away.

\paragraph{Working Memory Context.}
The \texttt{WorkingMemory} store $\mathcal{M}_t$ at step $t$ contains all
facts recorded in previous steps:

\[
\mathcal{M}_t = \bigcup_{i=1}^{t-1} \mathcal{F}_i,
\qquad
\mathcal{F}_i = \{(k, v) \mid \text{extracted from step } i\}.
\]

The formatted memory text is injected into the action prompt as:

\begin{lstlisting}[basicstyle=\small\ttfamily, breaklines=true,
                   frame=single, xleftmargin=10pt]
--- Your Working Memory ---
  [Step 3] symptom_count = 2
  [Step 5] booking_rule = >3 -> Specialist
Use these facts when making decisions.
\end{lstlisting}

\paragraph{Fact Parsing and Storage.}
After the LLM produces its output $O_t$, a regular expression extracts all
\texttt{[REMEMBER: ...]} blocks. Each comma-separated \texttt{key=value} pair
is parsed into a dictionary and appended to $\mathcal{M}$:

\[
\mathcal{F}_t = \texttt{parse\_re}(O_t, \texttt{\[REMEMBER:\s*(.*?)\]}).
\]

\subsubsection{Integration with PegaAgent}

LIDM is fully integrated into PegaAgent's \texttt{step()} loop. The prompt
assembly at each step is:

\[
P_t = P_t^{\text{T3A}} \oplus P_t^{\text{loop}} \oplus P_t^{\text{WMC}}
\oplus P_t^{\text{GAMP}},
\]

where $\oplus$ denotes string concatenation, $P_t^{\text{T3A}}$ is the base
T3A action prompt, $P_t^{\text{loop}}$ is the loop-detection notification
(if triggered), $P_t^{\text{WMC}}$ is the working memory context, and
$P_t^{\text{GAMP}}$ is the goal-aware memory instruction.

The \texttt{WorkingMemory} store is initialized at agent construction and
cleared on reset, ensuring no cross-task contamination. The mechanism adds
one lightweight LLM inference step per turn (the GAMP is part of the same
prompt, not a separate call), incurring negligible overhead.

\subsubsection{Why This Works for All LIDPs}

LIDM does not presuppose \textit{what} to remember, \textit{when} to remember
it, or \textit{when} to use it. The LLM determines all three dynamically:
\begin{itemize}[leftmargin=*,noitemsep,topsep=2pt]
    \item \textit{What:} The LLM identifies relevant facts based on the goal.
    \item \textit{When to record:} Every step triggers the GAMP, so any
    visible information can be captured.
    \item \textit{When to use:} The WMC is injected into every step's prompt,
    so recorded facts are always available for decision-making.
\end{itemize}

This design converts implicit, lossy memory (history summaries) into explicit,
structured facts that persist across app boundaries and step transitions---the
core capability required for long-chain implicit decision problems.



\subsection{Global Long-Horizon Planner}
The global planner is responsible for high-level task decomposition and long-range dependency modeling. It takes the task description and a lightweight GUI prior knowledge base as input and outputs a structured JSON plan.

\subsubsection{Input}
\begin{itemize}
    \item Task instruction (natural language)
    \item GUI prior knowledge base: Contains app functionality, common page hierarchies, and typical task flows (lightweight, token-efficient)
\end{itemize}

\subsubsection{Output}
A structured execution plan with the following fields:
\begin{itemize}
    \item \texttt{total_steps}: Estimated total number of actions
    \item \texttt{stages}: List of subtasks, each with:
    \begin{itemize}
        \item \texttt{id}: Unique subtask identifier
        \item \texttt{task}: Subtask description
        \item \texttt{depend}: List of prerequisite subtask IDs
        \item \texttt{checkpoint}: State verification condition for completion
        \item \texttt{cross\_app}: Boolean indicating if this subtask involves app switching
        \item \texttt{branch\_condition}: Optional conditional logic for branch points
    \end{itemize}
\end{itemize}

Example plan:
\begin{lstlisting}[language=json, caption={Example structured execution plan}]
{
  "total_steps": 28,
  "stages": [
    {"id": 1, "task": "Open Gallery app", "depend": [], "checkpoint": "Foreground app = Gallery"},
    {"id": 2, "task": "Select the latest photo", "depend": [1], "checkpoint": "Photo is selected"},
    {"id": 3, "task": "Share photo to WeChat", "depend": [2], "cross_app": true, "checkpoint": "Foreground app = WeChat"}
  ]
}
\end{lstlisting}

\begin{itemize}
    \item Only performs high-level logical decomposition, not low-level action generation, ensuring lightweight inference.
    \item Explicitly models conditional branches and long-range dependencies.

\subsection{Structured State Memory Module}
The memory module addresses the context explosion problem by storing only critical, structured information instead of raw screenshots or full a11y trees. It maintains three separate data stores:
\begin{enumerate}
    \item \textbf{Global Plan State}: Tracks current subtask progress, completion status, and anomalies.
    \item \textbf{Interface Critical State}: Stores structured summaries of checkpoint states (e.g., control states, text content, foreground app).
    \item \textbf{Trajectory Summary}: A condensed history of executed actions, with duplicate and invalid actions automatically filtered.
\end{enumerate}

\subsubsection{Dynamic Eviction Policy}
\begin{itemize}
    \item Completed subtasks: Retain only verification results, discard raw interface data.
    \item Short-term repeated actions: Automatically filtered to reduce redundancy.
    \item Only load data from the current stage and the most recent checkpoint into the LLM context, strictly limiting context length.
\end{itemize}

\subsection{Local Executor}
The local executor is responsible for executing individual subtasks. It is designed to be compatible with both a11y tree and pure vision approaches, allowing easy integration with existing state-of-the-art execution modules:
\begin{enumerate}
    \item \textbf{A11y Tree Version}: Parses the accessibility tree to locate elements and generate atomic actions.
    \item \textbf{Pure Vision Version}: Uses a vision-language model to understand the screenshot and predict action coordinates.
\end{enumerate}

\textbf{Constraint}: The local executor only operates within the scope of the current subtask and cannot modify the global plan, ensuring strict hierarchical decoupling.

\subsection{Anomaly Detection and Dynamic Replanning}
This module is the core of our error recovery mechanism. It performs real-time monitoring after each action execution and applies graded processing strategies based on anomaly severity.

\subsubsection{Anomaly Detection}
We detect four types of common long-horizon errors:
\begin{enumerate}
    \item \textbf{Action Failure}: Element is unclickable, text input does not match expected content.
    \item \textbf{Path Deviation}: Current interface does not match the expected state in the plan.
    \item \textbf{Loop Behavior}: Same action repeated N consecutive times (N = \textbf{[TO BE FILLED]}, configurable).
    \item \textbf{Asynchronous Interference}: Popups, loading pages, app crashes.
\end{enumerate}

\subsubsection{Graded Processing Strategy}
\begin{enumerate}
    \item \textbf{Mild Anomaly} (temporary popups, loading delays): Automatically close popups or wait intelligently, continue executing the current plan.
    \item \textbf{Moderate Deviation} (subtask execution failure): Roll back to the last valid checkpoint and retry the current subtask.
    \item \textbf{Severe Deviation} (unknown page, cross-app failure): Trigger global replanning, generating a new global plan based on the current interface state and original task.
\end{enumerate}

\subsubsection{Breakpoint Mechanism}
Breakpoints are automatically recorded after each subtask completion, cross-app transition, and critical operation. This allows arbitrary rollback to any valid state, preventing error propagation to subsequent steps.

\section{DISCUSSION}
\subsection{Advantages}
Our hierarchical planning approach addresses the key limitations of myopic GUI agents:
\begin{itemize}
    \item Explicit global planning eliminates goal forgetting in long sequences.
    \item Structured memory reduces context overhead and improves inference efficiency.
    \item Breakpoint backtracking blocks error propagation and improves robustness.
    \item Dual input compatibility supports both accessibility tree and pure vision scenarios.
\end{itemize}

\subsection{Limitations}
Our approach has several limitations that require future work:
\begin{itemize}
    \item Performance degrades on extremely long tasks (>50 steps) due to accumulated planning errors.
    \item Limited support for complex game interfaces and 3D applications where a11y trees are unavailable or incomplete.
    \item The current GUI prior knowledge base is manually constructed, limiting scalability to new apps.
\end{itemize}

\subsection{Future Work}
\begin{itemize}
    \item Extend the architecture to handle extremely long tasks (>100 steps) with multi-level planning.
    \item Integrate self-supervised learning to automatically build GUI prior knowledge from execution trajectories.
    \item Adapt the approach to desktop and iOS platforms to demonstrate generalizability.
\end{itemize}

\section{CONCLUSION}
In this paper, we presented a hierarchical long-horizon planning architecture for mobile GUI agents that addresses the critical limitations of existing myopic approaches. By integrating explicit global planning, structured state memory, and dynamic backtracking, our method achieves state-of-the-art performance on long-horizon tasks. We also introduced AndroidWorld-Long, a new benchmark for evaluating long-horizon GUI agent capabilities. Experimental results demonstrate that our approach significantly outperforms existing methods on complex tasks while maintaining strong performance on simple ones. Our work represents an important step toward building robust, general-purpose mobile GUI agents capable of handling real-world long-horizon workflows.
